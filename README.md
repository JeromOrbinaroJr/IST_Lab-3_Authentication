Командный проект: ИС «Виртуальный деканат».  
Стек: **Python + Flask + SQLite**.

Каждый участник отвечал за свою часть ЛР №3:
- **Часть 1** — аутентификация (постоянный пароль, хэширование)
- **Часть 2** — авторизация объектов (ACL)
- **Часть 3** — иерархическое ролевое управление доступом (RBAC)

## 2. Быстрый старт

### 2.1 Установка зависимостей

```bash
python -m venv .venv
source .venv/bin/activate      # Windows: .\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

### 2.2 Инициализация БД

```bash
python scripts/init_db.py   # создаёт instance/app.db со всеми таблицами
python scripts/seed_db.py   # заполняет пользователей, ACL и иерархию ролей
```

### 2.3 Запуск

```bash
python app.py
```

Открыть в браузере: `http://127.0.0.1:5000`

## 4. Схема БД

### Часть 1 — аутентификация

Таблица `users`:
- `username` — логин (unique)
- `password_hash` — хэш пароля `pbkdf2:sha256` (Werkzeug), пароль в открытом виде не хранится
- `role` — роль пользователя
- `is_active` — флаг активности

### Часть 2 — авторизация (ACL)

Таблица `records` — защищаемые объекты с владельцем (`owner_user_id`).

Таблица `acl` — правила доступа:
- `subject_type` — `role` или `user`
- `subject_value` — название роли или id пользователя
- `object_type` — тип объекта (например `records`)
- `action` — `read | read_own | edit_own | edit | full`

Иерархия действий: `full ⇒ edit ⇒ read`.

### Часть 3 — иерархия ролей (RBAC)

Таблица `roles` — справочник ролей с числовым уровнем.

Таблица `role_parents` — прямые связи наследования.

Таблица `role_permissions` — собственные права каждой роли.

## 5. Часть 1 — Аутентификация

**Выбранный вариант:** постоянный многократно используемый пароль.

**Маршруты:**
- `GET/POST /login` — вход по логину и паролю
- `POST /logout` — выход
- `GET /` — профиль (только после входа)
- `GET /admin/users` — список пользователей (только `admin`)
- `GET/POST /me/password` — смена пароля

**Реализация:**
- Хэширование паролей: `generate_password_hash` / `check_password_hash` (Werkzeug, алгоритм `pbkdf2:sha256`)
- Сессия: подписанная cookie Flask (`session`)
- Декораторы `login_required` и `roles_required(...)` в `app/security.py`

---

## 6. Часть 2 — Авторизация объектов (ACL)

**Выбранный вариант:** дискреционный способ с использованием списка прав доступа (Access Control List).

**Назначение прав для объектов типа `records`:**

| Роль     | Права                    |
|----------|--------------------------|
| student  | read_own, edit_own       |
| starosta | read_own, edit_own       |
| teacher  | edit                     |
| curator  | edit                     |
| dean     | full                     |
| admin    | full                     |

**Маршруты:**
- `GET /records/` — список записей
- `GET /records/<id>` — просмотр (по ACL + владение)
- `GET/POST /records/<id>/edit` — редактирование (по ACL + владение)
- `POST /records/<id>/delete` — удаление (только `full`)

**Реализация:** декораторы `acl_required`, `acl_read_own_required`, `acl_edit_own_required` в `app/security.py`.

---

## 7. Часть 3 — Иерархический RBAC

**Выбранный вариант:** иерархическое ролевое управление с наследованием прав.

**Иерархия:**

```
Студент (0) → Староста (1) → Преподаватель (2) → Куратор (3) → Деканат (4) → Администратор (5)
```

Каждая роль автоматически наследует все права ролей с более низким уровнем. Администратор обладает всеми правами системы.

**Маршруты:**
- `GET /rbac/` — обзор иерархии и таблица прав
- `GET /rbac/hierarchy` — данные из БД
- `GET /rbac/my-permissions` — права текущего пользователя с разбивкой по источнику
- `GET /rbac/area/<role>` — демонстрация проверки доступа к зоне
- `GET /rbac/grades` — пример `@rbac_required("teacher")`
- `GET /rbac/reports` — пример `@rbac_required("dean")`
- `GET /rbac/admin-panel` — пример `@rbac_required("admin")`

**Реализация:** модуль `app/rbac.py` — функции `has_role`, `get_inherited_roles`, `get_all_permissions` и декораторы `rbac_required`, `permission_required`.

## 8. Раздел «Студенты» — управление студентами и академическая адаптация (Осит Герман)

Реализовано локальное представление 1 из файла проекта БД: **«Управление студентами с системой академической адаптации»**.

### 8.1 Сущности ЛП1 (SQLite)

Сущности из таблиц 1.1/1.2/1.3 реализованы в `app/schema.sql`:
- `students` — студент (номер билета, ФИО, текущая группа, текущий статус)
- `academic_statuses` — справочник академических статусов
- `student_groups` — учебные группы (лимит мест + текущее число студентов)
- `teachers` / `disciplines` — преподаватели и дисциплины (`disciplines.is_advanced`)
- `engagement_scores` — оценка вовлеченности (1–10) + причина + дата
- `transfer_history` — история переводов (старая/новая группа, основание, сотрудник деканата)
- `academic_debts` — академические задолженности (активная/погашенная)

### 8.2 Страницы раздела

- `GET /students/` — список студентов (`admin/dean/teacher`)
- `GET/POST /students/new` — добавление студента (`admin/dean`)
- `GET /students/<id>` — карточка студента (`admin/dean/teacher`)
- `GET/POST /students/<id>/transfer` — перевод между группами (`admin/dean`)
- `GET/POST /students/<id>/debts/new` — добавить задолженность (`admin/dean`)
- `POST /students/debts/<id>/close` — погасить задолженность (`admin/dean`)
- `GET/POST /students/<id>/engagement/new` — добавить оценку вовлеченности (`teacher`, только по своим дисциплинам)

### 8.3 Мини‑правила (для демонстрации)

- контроль лимита мест: нельзя добавить/перевести студента в группу, где `current_students >= max_students`
- запрет перевода при активных долгах: если есть `academic_debts.status = 'активная'`, перевод блокируется

## 9. Раздел «Расписание и заметки» — организация образовательного процесса (ЛП2 - Жарков Даниил)

Реализовано локальное представление 2: **«Организация образовательного процесса и планирование»**.

### 9.1 Сущности (SQLite)

- `groups` — учебные группы для расписания (название, уникальное)
- `schedule` — расписание занятий (группа, день недели 1–6, время начала/конца, предмет, преподаватель, аудитория)
- `notes` — заметки студентов (привязаны к конкретной дате; личные или групповые)
- Поле `group_id` в `users` — привязка пользователя к группе расписания
- Поле `course` в `users` — курс студента (1–6), используется для фильтрации программ поддержки

### 9.2 Расписание

Студент и Староста видят расписание своей группы автоматически.  
Преподаватель и выше выбирают группу из выпадающего списка.

**Маршруты:**
- `GET /schedule` — расписание; по умолчанию открывается на сегодняшней дате (`?date=YYYY-MM-DD`)

**Возможности:**
- Выбор конкретной даты через `<input type="date">` — определяется день недели, показывается расписание
- Быстрый переход по дням текущей недели — ссылки ведут на конкретные даты (`?date=...`)
- При выборе воскресенья выводится сообщение «выходной день»

### 9.3 Заметки

**Маршруты:**
- `GET /notes` — список заметок с фильтром по дате
- `GET/POST /notes/new` — создать заметку (личную или групповую)
- `GET/POST /notes/<id>/edit` — редактировать свою заметку
- `POST /notes/<id>/delete` — удалить свою заметку

**Виды заметок:**

| Тип | Кто создаёт | Кто видит |
|-----|-------------|-----------|
| Личная | Студент, Староста | Только автор |
| Групповая | Только Староста | Все студенты своей группы |

**Интеграция с расписанием:**  
При просмотре расписания на конкретную дату под таблицей занятий отображаются заметки на этот день (личные + групповые от старосты). Кнопка `+ заметка` рядом с заголовком дня открывает форму с предзаполненной датой.

## 10. Раздел «Учебный план» — преподаватели, кафедры и дисциплины (ЛП3 - Кротков Павел Владиленович)

Реализовано локальное представление из файла проекта БД: «Назначение преподавателей на должности, кафедры и дисциплины».

Раздел сделан не как таблица БД, а как продуктовый экран веб-приложения: пользователь видит учебный план кафедр, распределение дисциплин, преподавателей, ставки, назначения и учебную нагрузку.

### 10.1 Сущности локального представления

Локальное представление реализовано в app/faculty.py.

Используются следующие сущности:

- DEPARTMENTS — кафедры:
  - Прикладная математика;
  - УИТС.
- TEACHERS — преподаватели:
  - ФИО;
  - учёная степень;
  - почта;
  - кафедра;
  - должность.
- DISCIPLINES — дисциплины:
  - название дисциплины;
  - преподаватель;
  - тип занятия;
  - часы;
  - кафедра.
- POSITIONS — справочник должностей:
  - профессор;
  - доцент;
  - старший преподаватель;
  - ассистент.
- QUALIFICATIONS — связь преподавателя с дисциплиной и уровнем подготовки.
- APPOINTMENTS — назначения преподавателей:
  - преподаватель;
  - должность;
  - дата назначения;
  - ставка.

### 10.2 Данные учебного плана

Кафедра Прикладная математика:

- Функциональный анализ - Холщевникова Наталья Николаевна

Кафедра УИТС:

- Базы данных - Бычков Сергей Юрьевич
- Основы проектирования и разработки Web-приложений - Подвигина Елена Анатольевна
- Машинное обучение и интеллектуальные системы - Ибатулин Михаил Юрьевич

### 10.3 Страницы раздела

- GET /faculty/ — просмотр учебного плана.
- POST /faculty/admin — сохранение изменений из админ-панели.

На странице GET /faculty/ отображаются:

- сводка по количеству преподавателей, кафедр, дисциплин и часов;
- карточки кафедр;
- дисциплины внутри каждой кафедры;
- карточки преподавателей;
- ставки и даты назначений;
- матрица готовности преподавателей;
- админ-панель для ролей dean и admin.

### 10.4 Права доступа

Иерархия демонстрационных пользователей:

student1 -> starosta1 -> teacher1 -> curator1 -> dean1 -> admin
Пользователи student1, starosta1, teacher1 и curator1 могут только просматривать учебный план.

Пользователи dean1 и admin получают доступ к админ-панели и могут редактировать:

- кафедры;
- дисциплины;
- преподавателей;
- кафедру преподавателя;
- должность преподавателя;
- дату назначения;
- ставку.

### 10.5 Мини-правила для демонстрации

- Пользователи ниже роли деканата не видят админ-панель.
- Сохранение изменений доступно только ролям dean и admin.
- Изменения из админ-панели сохраняются в файл:

instance/faculty_plan.json
- Если файл instance/faculty_plan.json отсутствует, приложение использует данные по умолчанию из app/faculty.py.
- Дисциплина может быть перенесена между кафедрами через админ-панель.
- Преподавателю можно изменить кафедру, должность, дату назначения и ставку.

### 10.6 Основные файлы реализации

- app/faculty.py — данные локального представления, маршруты, права доступа, сохранение изменений.
- templates/faculty/index.html — интерфейс учебного плана и админ-панели.
- templates/base.html — пункт меню «Учебный план».
- static/styles.css — стили карточек кафедр, преподавателей, матрицы готовности и админ-панели.
- app/__init__.py — регистрация blueprint faculty.


## 11. Раздел «Достижения, рейтинг и программы поддержки» — документальный контроль результатов обучения (ЛП4 - Данилин Кирилл)

Реализовано локальное представление 4: **«Документальный контроль успеваемости и аттестаций, результатов обучения»**.

### 11.1 Сущности (SQLite)

- `achievement_categories` — справочник категорий достижений (Олимпиада, Конференция, Спорт и др.) с базовым баллом
- `achievements` — достижения студентов: категория, название, описание, статус (`pending / approved / rejected`), баллы, рецензент, комментарий
- `events` — актуальные мероприятия (название, описание, дата, место, флаг активности)
- `program_types` — справочник типов программ поддержки (Университетская, Федеральная, Частный фонд)
- `programs` — стимулирующие программы с условиями: минимальный/максимальный курс, минимальный рейтинг, дедлайн, ссылка

### 11.2 Достижения

**Маршруты:**
- `GET /achievements` — студент видит свои достижения; преподаватель/admin видит все с фильтром по статусу
- `GET/POST /achievements/new` — студент подаёт достижение на проверку
- `GET /achievements/<id>` — детальная карточка
- `POST /achievements/<id>/review` — преподаватель выставляет баллы и одобряет/отклоняет

**Категории и базовые баллы:**

| Категория | Базовый балл |
|-----------|-------------|
| Олимпиада | 10 |
| Конференция | 5 |
| Спорт | 3 |
| Волонтёрство | 2 |
| Творчество | 2 |
| Сертификат/курс | 1 |
| Прочее | 1 |

### 11.3 Рейтинг

**Маршруты:**
- `GET /rating` — рейтинг

**Для студента/старосты:**
- Суммарный балл за все одобренные достижения
- Список зачтённых достижений с разбивкой по категориям
- Рейтинговая таблица своей группы с местом

**Для преподавателя/admin:**
- Общий рейтинг всех студентов с указанием группы и курса

### 11.4 Мероприятия

**Маршруты:**
- `GET /events` — список актуальных мероприятий
- `GET/POST /events/new` — добавить мероприятие (преподаватель и выше)
- `GET/POST /events/<id>/edit` — редактировать, скрывать/показывать

Студенты видят только активные мероприятия. Преподаватель может переключаться между активными и всеми.

### 11.5 Программы поддержки

**Маршруты:**
- `GET /programs` — список программ с фильтром по типу
- `GET/POST /programs/new` — добавить программу (преподаватель и выше)
- `GET/POST /programs/<id>/edit` — редактировать

**Автоматическое определение доступности** для студента — на основе курса и рейтинга:

| Статус | Условие                                  |
|--------|------------------------------------------|
| ✅ Вы подходите | Курс и рейтинг соответствуют требованиям |
| ⏳ Пока рано | Рейтинг ниже минимального                |
| (не показывается) | Курс превышает максимальный              |

**Типы программ:**
- Университетские — внутренние стипендии и гранты вуза
- Федеральные — государственные программы (стипендия Президента РФ, РФФИ и др.)
- Частный фонд / Благотворительность — гранты и стипендии от организаций
