# Виртуальный деканат — модуль аутентификации (Python + SQLite)

Моя часть - Авторизация объектов (ACL)

### Цель
Реализовать **авторизацию доступа к объектам** на основе модели **ACL (Access Control List)**: хранить правила доступа в БД и проверять их при обращении к защищаемым объектам.

### Рассмотренные модели (кратко) и выбор варианта
- **Матрица доступа**: плохо масштабируется — при большом числе субъектов/объектов становится громоздкой.
- **ACL (выбрано)**: гибко, масштабируется и удобно администрируется — права описываются набором правил.
  - пример: преподаватель → `edit`, студент → `read_own`/`edit_own`, деканат → `full`
- **Capability List**: сложно отзывать права и неудобно централизованно управлять доступом.
- **Lock–Key**: нестандартно для типового веб-приложения.
- **Мандатная модель**: избыточна для лабораторной.

### Реализация ACL в проекте

#### Защищаемые объекты
Для демонстрации объектной авторизации введён тип объектов **“записи/заметки”** (`records`) с владельцем.

#### Схема БД (ACL)
В `app/schema.sql` добавлены таблицы:
- `records` — защищаемые объекты (есть `owner_user_id`).
- `acl` — правила доступа:
  - `subject_type`: `role` или `user`
  - `subject_value`: название роли (например `student`) или id пользователя
  - `object_type`: тип объекта (например `records`)
  - `action`: `read | read_own | edit_own | edit | full`

#### Назначение прав (пример из задания)
В `scripts/seed_db.py` добавлено заполнение ACL для `object_type="records"`:
- `teacher` → `edit`
- `student` → `read_own` и `edit_own` (только свои записи)
- `dean` → `full`
- `admin` → `full` (для администрирования/демонстрации)

Также в БД создаются примерные записи `records` для демонстрации проверки прав.

#### Проверка прав в приложении
Проверка ACL выполнена через декораторы в `app/security.py`:
- `acl_required(object_type, action)` — проверка глобального права (иерархия: `full ⇒ edit ⇒ read`)
- `acl_read_own_required(...)` — “можно читать только своё”, если нет глобального `read/edit/full`
- `acl_edit_own_required(...)` — “можно редактировать только своё”, если нет глобального `edit/full`

#### Маршруты для демонстрации ACL
Добавлен blueprint `app/records.py` (подключён в `app/__init__.py`) и страницы в `templates/`:
- `GET /records/` — список записей
- `GET /records/<id>` — просмотр записи (по ACL + владение)
- `GET/POST /records/<id>/edit` — редактирование (по ACL + владение)
- `POST /records/<id>/delete` — удаление (только `full`)
